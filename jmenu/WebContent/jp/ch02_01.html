2.1 インターネットによるエンタープライズアプリケーションのインテグレーション (Internet Enterprise Application Integration)

インターネットは全世界の情報を連結し、新しい世界的なビジネスをもたらしました。インターネットによるエンタープライズアプリケーションのインテグレーションは益々不可欠なものなって来ました。JLIVE BuilderTMはEAIに最高なツールと環境を提供しています。過去のClient/Server形式と根本的に違います。古いコンセプトとしては必ずクロスネットワークからオープンネットワークにアップグレードにしなければなりませんでした。また、データの転送、セキュリティ等、ネットワーク的な考えを含めて全体解決方案(total solution)を立案しなければなりませんでした。

JLIVE BuilderTMはインターネットを考慮し(internet thinking)、また、解決する方式について最新の条件を充たしています。

　

2.11．インターネット的な考え (internet thinking)

  • 多階層Nーtier形式 (N-tiers architecture )

Nーtierはインターネットによるエンタープライズアプリケーションにとって絶対に必要な条件です。アプリケーションを実行する時、クライアントは直接データベースサーバにアクセスすることが許可されていません。必ずミードルウェアを経由し、企業内firewall内のデータベースにアクセスします。JLIVE BuilderTMはデフォルトで少くなくでも3−tier構造になっています。ミドル層はJAVATMでServletで構成され、どんなプラットフォームに適応しています。

  • 集中アプリケーション構造、分散式な処理モジュール構造 (centralized programs and distributed computing)

インタネットクライアントは世界中のどこにでもあり、昔のように必ずアプリケーションを全部のクライアントにインストールする必要がありません、インタネットアプリに分散モジュールをWebはサーバに集中します。クライアントからサービスとモジュール実行のリクニストがあった時、Webは今クライアントが必要となる小さなモジュールだけを転送すれば結構です、アプリケーション全部を転送する必要はありません。もし、このモジュールが既にクライアントに存在する場合、直接起動ができます。インターネットアプリを実行するのがブラウザは表示機能だけではなくフォーミュラ処理機能ももっています。JLIVE BuilderTMはJavaTM Applet Swing とclass loaderなどのコンポーネントを用しているので、インターネットアプリケーションの開発、実行、メンテナンス、システム全体の分散式処理ができます。つまり、クライアントからのリクニスト、ミドル層、サーバ、役割分担で効率よく処理されるようになっています。

  • プラットフォームにこだわりなし (for all platforms)

インターネットアプリは、通常、各種ブラットフォームブラウザで実行されます。その為、アプ
リの第一条件は、必ず各種類のブラットフォームの上で実行できることです。JLIVE BuilderTMで使用しているJavaTMが当然一番良い選択肢となっています。

  • データベースにこだわりなし(for all databases)

インターネットによるエンタープライズアプリケーションのインテグレーションを行うにあたり、企業内部には、新旧データベースを同時に使用している可能性があります、ミドル層はクライアントからみると必ず各種のデータベースにアクセス出来るようにしなければなりません。JLIVE BuilderTMのミドルウェアは同時に各数の種類別のデータベースにアクセスをサポートしているミドルウェアです。

  • 権限を授けられたセキュリティコントロール (authorized security control)

インターネットは不完全な実行環境にあり、ユーザに対し、授権検査、認証検査のステップをミドルウェアで実行し、システムの安全性を確保しています。ユーザ認証インフォーメーションがデ
ータベースサーバに保存されています。管理者はニーズによって権限を分配されていす。JLIVE BuilderTMはデフォルトのセキュリティ コントロールシステムを提供し、開発者はニーズによって修正することができます。

  • データの転送量とサーバーローティング (data transferring and server loading)

インターネットNーtierシステムの場合、クライアントの情報はインターネットを通して獲得
されます。そこで、インターネットと内部実行スピード違いがある為、対応を考えなければなり
ません。JLIVE BuilderTMはデフォルトVirtual　モードを持ち、Request on demand方式を利用することにより、転送ボリュームにかかわらず、システムはクライアントの最小のリクエストに対し、Background treadで分割して転送します。その為、処理結果のボリュームに関係なく、クライアントへの反応時間はほぼ同じです。しかも、Request on demandが独自に実行しているコンポーネントにコントロールされている為、特別な開発者からのメンテナンスは必要ありません。

JLIVE BuilderTM中のデータソース、リストボックス、Combo box、Gridなどのコンポーネントはこの機能を生かしています。

  • アプリケーションの流れとユーザインターフェース（system flow and user interactive)

インターネットによるエンタープライズアプリケーションのインテグレーションは、単純或いはダイナミックなホームページでは実現できません。ホームページにはシステムフローがありません。ホームページは一般に会社か商品の説明に使用されていますが、インターネットアプリケーションはC/S方式と同様に、システム上、ユーザインタフェースで即時にユーザに対してサービス反応し、連動して作業できます。

  • インターネットレポートシステム (internet report system)

レポート業務はエンタープライズアプリケーションにとって不可欠のサービスです。使用上最小限の条件として：

  • Query　データ印刷(query printing data)

アプリレポート作業において、通常ユーザクエリ-後にデータ印刷作業が行われます。その為、レポート業務とユーザインターフェースは統合されなければなりません。

  • ページ分け (paging)

例えば、ユーザは200ページの資料をプリントする必要がある場合、システムは必ずユーザ側でレポートの寸法によってページ分割し、ユーザが全部印刷するか部分にするかを決め、印刷機能をクリックし実行させます。この機能はWindowsアプリに良く利用されていますが、インターネットHTMLでは実現できません。ユーザ側でページ分けプリント作業リスティングがなければ、すべてユーザ側で作業を行うと、負担がかかり過ぎますので、アプリ実行ができません。

  • ｲﾒｰｼﾞと統計チャート (image and statistic chart)

レポート業務において、ｲﾒｰｼﾞと統計チャートが不可欠です。JLIVE BuilderTMの場合はクライアント側で即時に描かれサーバに負担をかけません。インターネットアプリなので、レポート業務はどんなプリンターにおいて同じく印刷されなければなりません。JLIVE BuilderTMレポート設計の時、絶対位置を設計し、見た目と印刷物は一致しています。

  • 絶対的なプリント位置 (absolute printing position)

インターネットでプリントサービス提供する場合，システムはどんな種類のプリンターに置いても同じのレイアウト・位置を正確にアウトプットしなければなりません。jLIVE Builder™ レポーティング設計の時，絶対位置 (absolute printing position) を利用して配置することになっています。レイアウトに見えているものがイコール印刷で得られるものとなっています (what you see is what you get)。

  • リアルタイム印刷 (real time printing)

JLIVE BuilderTMレポートシステムはリアルタイムがあり、プリントした資料は最新の情報でクライアントとサーバの効率を最適にされるようにされています。

レポートシステムの場合、クライアントのリクニストに基いて結果集を作りますだけで、このような業務分けになります。よって、インターネットレポートシステムがスムーズに行われています。
